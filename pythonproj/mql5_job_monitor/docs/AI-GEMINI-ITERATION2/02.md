### **Improvements for `USER_MANUAL.md`**

This manual is clear and user-friendly. The following suggestions are aimed at protecting the user and clarifying some of the more technical steps.

#### 1. **Critical Security Warning on Credential Handling**

The current manual states that "Credentials are stored locally in your browser." As noted previously, this is a significant security risk.

*   **Recommendation:** Modify the application to **not** store credentials in the browser's local storage. Instead, the credentials should be held in memory by the server only for the duration of the scraping session.

The User Manual should be updated to reflect this more secure approach:

*   **In Section 4, "Configuration Options,"** remove the note about local storage.
*   **In Section 3, "Step 4: Configure Your Settings,"** add a new security note:
    > **Security Note:** Your MQL5 credentials are used directly for the scraping session and are **never** saved in your browser or on the server. You will need to have the settings panel open with your credentials entered when you click "Scrape New Jobs."

#### 2. **Clarify Data Deletion Process**

The manual suggests deleting the `app.db` file to clear old data. This is a crude but effective method. A more user-friendly approach would be better.

*   **Recommendation:** The `cleanup_old_jobs` function from the `DEPLOYMENT_GUIDE.md` should be wired up to a button in the UI, such as "Clean Up Old Jobs" within the Settings panel.
*   The "Data Management" section in the manual can then be updated:
    > **Clean database periodically:** To save space and improve performance, you can remove jobs older than 30 days. Click the Settings button (⚙️) and select "Clean Up Old Job Data."

#### 3. **Refine Startup Instructions**

*   **Recommendation:** As previously suggested, rename `test_app.py` to `run.py` or `app.py` to avoid confusion. The manual should be updated to reflect this change in all `python test_app.py` commands.

### **Improvements for `DEPLOYMENT_GUIDE.md`**

This is an excellent guide for a sysadmin. The suggestions below focus on production-readiness and maintainability, building on the strong foundation you've already established.

#### 1. **Use a Production-Ready WSGI Server**

The guide's `service_app.py` script still uses Flask's development server (`app.run`), which is explicitly not suitable for production.

*   **Recommendation:** Use a production-ready, pure-Python WSGI server like **`waitress`**, as it's Windows-friendly.

    1.  Add `waitress` to your `requirements.txt` file.
    2.  Modify the `service_app.py` script:
        ```python
        # service_app.py
        import os
        from waitress import serve
        # Assumes the main app is in 'run.py' and the Flask object is 'app'
        from run import app 
        
        # Change host to '127.0.0.1' to only allow local connections
        # Change to '0.0.0.0' to allow network connections (ensure firewall is configured)
        serve(app, host='127.0.0.1', port=8080)
        ```

#### 2. **Centralize Configuration with `.env` Files**

The guide mentions changing the secret key directly in the code and uses hardcoded paths. This can be made more secure and flexible.

*   **Recommendation:** Use a `.env` file for all configuration.
    1.  Create a file named `.env.example` and add it to your repository:
        ```
        # Flask secret key (change this to a long random string)
        SECRET_KEY='your-super-secret-key-here'
        
        # Database URI
        DATABASE_URI='sqlite:///src/database/app.db'
        
        # Log file path
        LOG_FILE='logs/mql5_monitor.log'
        ```
    2.  Instruct administrators to copy `.env.example` to `.env` and populate it. Add `.env` to your `.gitignore` file.
    3.  Use a library like `python-dotenv` to load these settings into your Flask application at startup.

#### 3. **Make Scripts More Portable**

The `start_app.bat` script uses a hardcoded path (`C:\MQL5JobMonitor`). This makes the application difficult to move.

*   **Recommendation:** Modify the script to use relative paths based on its own location.
    ```batch
    @echo off
    rem Get the directory of the batch file
    cd /d "%~dp0"

    echo Starting MQL5 Job Monitor from %CD%...
    
    rem Check for venv
    if not exist "venv\Scripts\activate.bat" (
        echo Virtual environment not found. Please run the setup first.
        pause
        exit /b
    )
    
    call venv\Scripts\activate
    python run.py  rem <-- Assumes you renamed test_app.py
    pause
    ```

#### 4. **Integrate Maintenance Scripts with Flask-CLI**

Running separate `backup_db.py` or `cleanup_old_jobs.py` scripts is good, but integrating them into the application itself provides a more unified and maintainable solution.

*   **Recommendation:** Use Flask's Command Line Interface (CLI) to create custom commands.
    *   In your main application file (`run.py`), you can define functions that can be called from the command line:
        ```python
        # In run.py or a new commands.py file
        import click

        @app.cli.command("cleanup-jobs")
        @click.option('--days', default=30, help='Delete jobs older than this many days.')
        def cleanup_jobs_command(days):
            """Cleans up old job postings from the database."""
            # Your cleanup logic here
            print(f"Cleaned up jobs older than {days} days.")

        @app.cli.command("backup-db")
        def backup_db_command():
            """Creates a timestamped backup of the database."""
            # Your backup logic here
            print("Database backup created.")
        ```
    *   The Windows Task Scheduler would then be configured to run these commands:
        ```bash
        # Example for the cleanup task
        schtasks /create /tn "MQL5 Job Monitor Cleanup" /tr "C:\MQL5JobMonitor\venv\Scripts\flask.exe cleanup-jobs --days=30" /sc daily /st 03:00
        ```
