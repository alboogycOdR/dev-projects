### **CRITICAL SECURITY VULNERABILITY: Plaintext Credential Storage**

This is the single most important issue to fix in the entire application.

*   **The Problem:** The `saveSettings` and `loadSettings` functions use `localStorage` to store and retrieve the user's MQL5 username and password. `localStorage` is a client-side storage mechanism that saves data in **plaintext**.

*   **The Risk:**
    1.  **Cross-Site Scripting (XSS):** If your application ever has an XSS vulnerability (e.g., from a job description or a third-party library), a malicious script could easily steal the credentials with `localStorage.getItem('mql5JobMonitorSettings')`.
    2.  **Browser Inspection:** Anyone with access to the user's computer can open the browser's developer tools, go to the "Application" tab, select "Local Storage," and see the username and password in clear text.
    3.  **Physical Access:** If the user's computer is compromised, these saved credentials are trivial to find.

*   **How to Fix:** **Never store passwords on the client-side.** Credentials should be entered by the user and held in memory *only* for the duration of the API request.

#### **Recommended Changes:**

1.  **Modify `saveSettings()`** to no longer save the username or password. It can continue to save non-sensitive settings like `maxPages`.
2.  **Modify `showSettings()`** to *never* pre-populate the password field. It is acceptable to pre-populate the username.
3.  **Modify `scrapeJobs()`** to get the credentials directly from the form inputs *at the moment of the request*.

Here is the corrected code for each function:

**1. Corrected `saveSettings()`**
```javascript
// Remove password from being saved. Username is optional.
saveSettings() {
    this.settings.username = document.getElementById('usernameInput').value;

    const settingsToStore = {
        username: this.settings.username,
        maxPages: this.settings.maxPages
    };

    localStorage.setItem('mql5JobMonitorSettings', JSON.stringify(settingsToStore));

}
```

### **Code Quality and User Experience Improvements**

These are less critical than the security flaw but will improve your code's maintainability and robustness.

#### 1. Use Event Delegation for Dynamic Elements

*   **Problem:** In `renderJobs`, you add an `onclick="app.markAsRead(...)"` attribute to every "Mark as read" button. This creates many separate event listeners and mixes JS logic into your HTML string.
*   **Solution:** Use event delegation. Add a single click listener to a static parent container (e.g., `#jobsList`) and determine which button was clicked from the event object.

**Recommendation:**

```javascript
// In init() or bindEvents()
// 1. Add ONE listener to the parent
document.getElementById('jobsList').addEventListener('click', (e) => {
    // Check if the clicked element (or its parent) is a "mark as read" button
    const markButton = e.target.closest('.mark-as-read-btn');
    if (markButton) {
        e.preventDefault(); // Stop any default button action
        const jobId = markButton.dataset.jobId; // Get the ID from a data attribute
        this.markAsRead(jobId);
    }
});

// In renderJobs()
// 2. Add a class and data-attribute instead of onclick
// Note the new class `mark-as-read-btn` and `data-job-id="${job.id}"`
container.innerHTML = jobs.map(job => `
    //... (rest of the job card) ...
    <div class="flex items-center justify-between mt-4">
        <span class="text-xs text-gray-500">Added: ${new Date(job.date_added).toLocaleDateString()}</span>
        ${job.is_new ? `<button data-job-id="${job.id}" class="mark-as-read-btn text-xs text-blue-600 hover:text-blue-800">Mark as read</button>` : ''}
    </div>
`).join('');
```

#### 2. Separate HTML Structure with `<template>`

*   **Problem:** The `renderJobs` and `renderError` methods contain large, hard-to-read blocks of HTML inside JavaScript template literals.
*   **Solution:** Use the `<template>` element in your `index.html` to hold the structure. Your JavaScript can then clone this template, populate it with data, and append it to the DOM. This provides a clean separation of concerns.

**Recommendation:**

**In `index.html`:**
```html
<template id="jobCardTemplate">
    <div class="job-card bg-gray-50 rounded-lg p-6 mb-4 border border-gray-200 fade-in">
        <!-- ... the full structure of your job card ... -->
        <h3 class="job-title"></h3>
        <!-- ... etc. -->
    </div>
</template>
```

**In `app.js` (`renderJobs` method):**
```javascript
renderJobs(jobs) {
    const container = document.getElementById('jobsList');
    const template = document.getElementById('jobCardTemplate');
    container.innerHTML = ''; // Clear previous jobs

    if (jobs.length === 0) { /* ... render no jobs message ... */ }

    const fragment = document.createDocumentFragment();
    jobs.forEach(job => {
        const clone = template.content.cloneNode(true);
        // Use querySelector on the cloned fragment to find and populate elements
        clone.querySelector('.job-title').textContent = job.title;
        // ... populate price, description, etc.
        fragment.appendChild(clone);
    });
    container.appendChild(fragment); // One single DOM update
}
```
