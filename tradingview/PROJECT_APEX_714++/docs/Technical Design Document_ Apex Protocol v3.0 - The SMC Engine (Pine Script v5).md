# Technical Design Document: Apex Protocol v3.0 - The SMC Engine (Pine Script v5)

**Document Version:** 1.0
**Date:** August 31, 2025
**Author:** Manus AI

## 1. Introduction

This document details the technical design and implementation plan for the Apex Protocol v3.0, also known as "The SMC Engine," a modular, multi-strategy TradingView indicator built on Smart Money Concepts (SMC). The system will be developed using Pine Script v5, adhering to the specifications outlined in the "Technical Specification: Apex Protocol v3.0 - The SMC Engine" document (Version 3.1, August 31, 2025).

## 2. Core Architectural Principles for Pine Script v5

The core philosophy of separating the **Analysis Engine** from the **Strategy Logic** will be maintained. In Pine Script, this translates to a modular code structure using functions and libraries where appropriate. The primary script will act as the orchestrator, calling functions from the SMC Engine to identify POIs and then passing this data to the selected Strategy Modules.

### 2.1 SMC Engine (POI Detection)

The SMC Engine will consist of a collection of functions, each responsible for detecting a specific Smart Money Concept Point of Interest (POI). These functions will be designed to be robust, efficient, and reusable across different strategy modules. Given Pine Script's execution model, these functions will primarily operate on historical bar data and real-time bar updates.

**Key Considerations:**
*   **Function Modularity:** Each POI detection (e.g., `BOS`, `FVG`, `OB`, `Liquidity Sweep`, `SMT Divergence`) will be encapsulated in its own function. This promotes readability, maintainability, and testability.
*   **Multi-Timeframe (MTF) Data Handling:** The `request.security()` function will be crucial for fetching data from higher timeframes (HTF) and correlated assets without repainting. Careful management of `request.security()` calls will be necessary to optimize performance and avoid excessive resource usage.
*   **State Management:** Pine Script's `var` keyword will be utilized to maintain the state of detected POIs across bars, ensuring that POIs are correctly identified and tracked over time. This is particularly important for concepts like `BOS` and `MSS` which depend on previous bar states.

### 2.2 Strategy Modules

Each strategy module (e.g., "The Core SMC Model," "ICT Silver Bullet") will be implemented as a distinct block of logic within the main Pine Script. These modules will consume the POI data generated by the SMC Engine and apply their specific rules to identify trade setups. User input via indicator settings will determine which strategy modules are active for specific Killzones.

**Key Considerations:**
*   **Conditional Execution:** Strategy logic will only execute when the indicator is within the specified Killzone and the corresponding strategy module is selected by the user.
*   **Sequential Event Processing:** The strategies, especially "The Core SMC Model," require a sequence of events. This will be managed using state variables (`var` keyword) to track the progress through the event checklist.
*   **Alerting:** A single `alertcondition()` will be used, triggered only when a strategy module reaches the `ARMED` state. The alert message will be dynamically generated to indicate the triggered strategy and Killzone.

### 2.3 State Machine

The global State Machine (`STANDBY`, `HUNTING`, `CONFIRMING`, `ARMED`) will be implemented using `var` variables to track the current state of the indicator. Transitions between states will be governed by the Killzone times and the progression of events within the selected strategy modules.

### 2.4 User Interface (UI)

Pine Script's built-in input functions (`input.bool`, `input.string`, `input.float`, `input.int`, `input.timeframe`, `input.dropdown`) will be used to create the indicator settings. The on-chart dashboard will be rendered using `label.new()` and `table.new()` functions to display the protocol status, HTF bias, and the strategy checklist.

**Key Considerations:**
*   **Input Organization:** Inputs will be logically grouped using `group` and `tooltip` arguments within the input functions for a clean and user-friendly interface.
*   **Dynamic Dashboard:** The on-chart dashboard will be updated on each bar to reflect the current state of the system and the progress of the active strategy.

## 3. Detailed Implementation Plan (Phase 1: SMC Engine POI Detection)

This section outlines the detailed plan for implementing the SMC Engine's POI detection functions. Each POI will have a dedicated function, and where necessary, helper functions will be created.

### 3.1 Market Structure

#### 3.1.1 Break of Structure (`BOS`)

**Definition:** A candle closes decisively above a validated swing high (bullish) or below a validated swing low (bearish) in alignment with the Higher Timeframe trend.

**Implementation:**
*   **Swing High/Low Detection:** A helper function `f_isSwingHigh(length)` and `f_isSwingLow(length)` will be implemented to identify validated swing points. This typically involves comparing a bar's high/low with a certain number of preceding and succeeding bars.
*   **Decisive Close:** A `close` above/below the swing point, potentially with a configurable buffer or percentage.
*   **HTF Trend Alignment:** This will require fetching HTF data using `request.security()` and determining the HTF trend (e.g., based on moving averages or HTF market structure).
*   **Function Signature:** `f_detectBOS(swingHighPrice, swingLowPrice, htfTrend)`
*   **Output:** Returns `true` if BOS is detected, along with the type (bullish/bearish) and the price level.

#### 3.1.2 Market Structure Shift (`MSS` / `CHoCH`)

**Definition:** The first `BOS` that occurs against the prevailing trend, signaling a potential reversal. Coded as `Change of Character`.

**Implementation:**
*   Builds upon the `BOS` detection. Requires tracking the prevailing trend. When a `BOS` occurs in the opposite direction of the prevailing trend, it's an `MSS`.
*   **Function Signature:** `f_detectMSS(prevTrend, currentBOS)`
*   **Output:** Returns `true` if MSS is detected, along with the new potential trend direction.

### 3.2 Liquidity

#### 3.2.1 Session Liquidity (Asian H/L, London H/L)

**Definition:** High and Low of specific trading sessions.

**Implementation:**
*   Use `time()` and `time_close()` functions with `session.regular` or custom session strings to define session boundaries.
*   `request.security()` will be used to get the high and low of the previous session.
*   **Function Signature:** `f_getSessionHighLow(sessionString, lookbackBars)`
*   **Output:** Returns the high and low prices for the specified session.

#### 3.2.2 Structural Liquidity (PDH/PDL, PWH/PWL, Swing H/L)

**Definition:** Highs and lows of previous days/weeks, and validated swing highs/lows.

**Implementation:**
*   `PDH/PDL, PWH/PWL`: Use `request.security()` with `timeframe.daily` and `timeframe.weekly` respectively, combined with `high` and `low` series.
*   `Swing H/L`: Re-use the `f_isSwingHigh()` and `f_isSwingLow()` helper functions from the Market Structure section.
*   **Function Signature:** `f_getStructuralLiquidity()`
*   **Output:** Returns a collection of relevant liquidity levels.

### 3.3 Price Imbalances & Key Zones

#### 3.3.1 Fair Value Gap (`FVG`)

**Definition:** A three-candle pattern creating an imbalance. Must detect on all user-relevant timeframes (e.g., M5, M1, 30s).

**Implementation:**
*   **Three-Candle Pattern:** `high[1]` < `low[3]` for bullish FVG, `low[1]` > `high[3]` for bearish FVG.
*   **MTF Detection:** The FVG detection function will accept a timeframe argument and use `request.security()` to fetch data from that timeframe.
*   **Function Signature:** `f_detectFVG(timeframe)`
*   **Output:** Returns `true` if FVG is detected, along with its top and bottom price levels.

#### 3.3.2 Order Block (`OB`)

**Definition:** The last opposing candle before a strong, structure-breaking price move.

**Implementation:**
*   Requires identifying a strong move (e.g., large candle body, significant price displacement) and the preceding opposing candle.
*   **Function Signature:** `f_detectOB(isBullishMove)`
*   **Output:** Returns `true` if OB is detected, along with its high and low price levels.

#### 3.3.3 Inversion Fair Value Gap (`IFVG`)

**Definition:** A Fair Value Gap that is first disrespected and traded through, then re-offered as a new level of support/resistance.

**Implementation:**
*   Builds upon `FVG` detection. Requires tracking if an `FVG` has been traded through and then re-entered.
*   **Function Signature:** `f_detectIFVG(fvgPriceLevels)`
*   **Output:** Returns `true` if IFVG is detected, along with its new support/resistance levels.

### 3.4 Correlated Asset Divergence

#### 3.4.1 Smart Money Technique (`SMT`) Divergence

**Definition:** Requires a secondary data feed (e.g., `TVC:DXY`). The Engine must detect divergences where the primary asset makes a new high/low, but the correlated asset fails to confirm it.

**Implementation:**
*   Use `request.security()` to fetch data for the correlated asset (e.g., `TVC:DXY`).
*   Compare the highs/lows of the primary asset with the correlated asset. A divergence occurs when one makes a new extreme, but the other does not.
*   **Function Signature:** `f_detectSMTDivergence(correlatedAssetTickerID)`
*   **Output:** Returns `true` if SMT Divergence is detected, along with the type (bullish/bearish).

## 4. Detailed Implementation Plan (Phase 2: Strategy Modules)

This section outlines the detailed plan for implementing the strategy modules.

### 4.1 Module A: "The Core SMC Model"

**Implementation:**
*   **State Variables:** Use `var` variables to track the completion of `Event 1`, `Event 2`, `Event 3`, and `Event 4`.
*   **Event 1 (Manipulation Confirmed):** Call `f_getSessionHighLow()` and `f_getStructuralLiquidity()` for Liquidity Sweep detection. Call `f_detectSMTDivergence()` for SMT Divergence. Set `event1Confirmed = true` if either is true.

### 4.2 Module B: "ICT Silver Bullet" (Simplified Model)

**Implementation:**
*   **Primary Condition:** Check if the current time is within the specified window (e.g., 10:00-11:00 EST) using `time()` and `hour()`/`minute()` functions.
*   **Event 1 (Liquidity Sweep):** Call `f_getSessionHighLow()` or `f_getStructuralLiquidity()` to detect a liquidity sweep of a recent session or swing high/low. Set `event1Confirmed = true`.
*   **Event 2 (Price Displacement & FVG):** After `event1Confirmed`, detect a strong price displacement that creates a clean Fair Value Gap (`FVG`). This will involve calling `f_detectFVG()` on the current timeframe. Set `event2Confirmed = true`.
*   **ARMED State:** If both events are confirmed, transition the State Machine to `ARMED` and trigger the alert. The entry will be the detected FVG.

## 5. Detailed Implementation Plan (Phase 3: State Machine & User Interface)

### 5.1 State Machine Logic

**Implementation:**
*   A `var` string variable, `protocolStatus`, will hold the current state (`STANDBY`, `HUNTING`, `CONFIRMING`, `ARMED`).
*   **`STANDBY`**: Default state. Transitions to `HUNTING` when `time` enters a selected Killzone.
*   **`HUNTING`**: When `time` is within a selected Killzone. Actively monitors for `Event 1` of the selected strategy. Transitions to `CONFIRMING` upon `Event 1` confirmation.
*   **`CONFIRMING`**: After `Event 1` (and subsequent events for Core SMC Model) are confirmed. Waits for the final confirmation event. Transitions to `ARMED` upon final event confirmation.
*   **`ARMED`**: All sequential events of the selected Strategy Module have been confirmed. Triggers a single alert and plots trade visuals. Stays `ARMED` until the end of the Killzone or a new bar starts, then resets to `STANDBY` or `HUNTING` for the next Killzone.

### 5.3 UI Design - On-Chart Dashboard

**Implementation:**
*   A `table.new()` object will be created once at the beginning of the script to serve as the dashboard container. Its position will be fixed (e.g., `table.new(position.top_right, ...)`).
*   `table.cell()` will be used to populate the table with text and potentially background colors for status indicators.
*   `label.new()` will be used for the checklist items, dynamically updated with checkmark symbols (e.g., `\u2713` for a checkmark) or empty boxes based on the state variables of the active strategy.
*   The dashboard will be updated on each bar using `table.cell_set_text()` and `label.set_text()` to reflect the current `protocolStatus`, `HTF Bias`, and the progress of the selected strategy's checklist.

## 6. Alerting Mechanism

**Implementation:**
*   A single `alertcondition()` will be defined. This condition will be `true` only when the `protocolStatus` variable transitions to `ARMED`.
*   The `message` argument of `alertcondition()` will be a dynamic string constructed using `str.format()` to include the triggered strategy module and the Killzone (e.g., "APEX: {0} \'{1}\' {2} Signal!").
*   Example `alertcondition`:
    ```pinescript
    alertcondition(protocolStatus == "ARMED" and ta.change(protocolStatus), 
                   message = str.format("APEX: {0} '{1}' {2} Signal!", 
                                      activeKillzone, activeStrategyName, 
                                      (isBullishSignal ? "BUY" : "SELL")), 
                   title = "Apex Protocol v3.0 Alert")
    ```
